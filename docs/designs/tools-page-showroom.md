# CEO Plan: Tools Page — Self-Updating Showroom

## Problem

The /tools page silently contradicts the site's core claim ("six bots build this
daily"). The interactive half is frozen: the model roster is a hardcoded
`TRACKED_IDS` list in `bot/model_data_bot.py` (lines 57-79) untouched since
2026-02-23 (no Fable 5, no Opus 4.7, no Gemini 3.5-family models — Fable 5 was
featured on the site's own Radar TODAY, Gemini 3.5 Live Translate also on today's Radar),
the featured spotlight has shown only Chat Playground since the spotlight shipped (2026-03-13),
write-ups render oldest-first with no dates (`src/pages/tools/index.astro:8`
has no sort), tool_bot fabricates hands-on testing claims, and status badges
are write-once decoration.

Landscape: model-comparison is saturated (Artificial Analysis 311+ models 8x/day,
PricePerToken 300+ daily). SOFT CAT cannot and should not compete on breadth.
Differentiation = the machinery itself, made visible. Freshness-as-product.

## Vision

### 10x Check
The Tools page as the live showroom of the pipeline: every datum shows its
provenance ("refreshed by model_bot 06:00 today"), the roster grows itself via
bot-proposed PRs (horizon_bot pattern: 18 proposal PRs opened since 2026-04-22,
8 merged — the ~55% rejection rate is a design input, see B4), decay is visible
and self-healing (dead links auto-archive write-ups after a grace window). A
visitor can catch the machinery WORKING, never catch it dead.

## Implementation Specifics (resolved ambiguities)

- **B1 date source (D13):** one-time script writes explicit `date:` frontmatter
  into all 39 write-ups (36 from filename prefix, 3 undated ones from a one-time
  git first-commit lookup); schema field REQUIRED thereafter; tool_bot template
  adds the line going forward. Sort: date desc. No build-time derivation.
- **B2 scrub:** one-time audit of all 39 write-ups for first-person testing
  claims ("we tested", "we tried", "we've been using"); rewrite to attributed
  or neutral phrasing. At least 3 affected: 2026-05-31-hermes, 2026-03-22-liteparse,
  2026-03-01-sakana. Ships in PR 1 alongside the prompt rule.
- **B3 score source:** capability scores assigned manually by Valori, calibrated
  against Artificial Analysis benchmark data and verified against live sources
  (per the project's verify-bot-output rule). The bot's `NEW_MODEL_DEFAULTS`
  placeholders are never auto-published.
- **B4 trigger (decided D4, E4):** RADAR-GATED. Propose a model only when it
  (a) is listed on OpenRouter, (b) was featured on the site's own Radar, and
  (c) its id is not already in models.json. Radar-to-OpenRouter join =
  deterministic normalized-name match (lowercase, strip non-alphanumerics,
  bidirectional substring), a pure function with a table-driven pytest. The PR body cites the radar entry + OpenRouter id.
  Expected volume ~1-2 PRs/month. Same review surface as horizon_bot Job 1. Honest expectation set by horizon data: roughly half get rejected;
  that is the system working, not failing. Max 1 open roster PR at a time;
  new proposals push as commits to the existing open PR branch (preserves
  review discussion); a new PR opens only when none is open. (D11)
- **B5 rotation (D14):** if any manifest tool has `featured: true` it is pinned
  (editorial override); otherwise spotlight = ISO week number mod count of ALL
  manifest tools. Flag becomes opt-in, not removed. Pure function of date +
  manifest — deterministic, testable, no state.
- **E5 trackedSince (renamed per D15):** B3's refresh adds `trackedSince` to
  every models.json entry (existing 21 backdated — no badge flood on day one);
  B4's PRs set the proposal date. Badge = "newly tracked", shown when
  `trackedSince` is within 14 days of build.
- **B6/E1 freshness display (amended D15):** absolute ISO timestamp rendered
  SERVER-SIDE as visible text (crawlers/no-JS/screenshots keep the proof); a
  tiny inline script upgrades it to relative age client-side. A failed deploy
  never shows a false "today" — displayed age grows until a deploy succeeds.
- **E2 matching:** no fuzzy matching. Optional `radarRef: "YYYY-MM-DD#entry-id"`
  field in models.json, populated manually in B3 and by B4 PRs. ROT GUARD (D15):
  radar day-files are pruned as they age out of the manifest (PR #143), so the
  build checks the day-file exists and degrades the link to plain text when
  pruned; validate-tools-data.mjs checks all radarRefs resolve or are flagged.
- **E3 guard + write path (amended D12):** a write-up's `url` must return a
  definitive-dead signal (404/410/DNS-fail/conn-refused) on 3 CONSECUTIVE
  weekly checks before status flips to `archived`.
  Failure-streak state persists in `bot/tool_verify_history.json` (same
  history-file pattern as `bot/radar_history.json`). Status flips +
  `last_verified` stamps are committed directly by tool_bot (same
  direct-commit pattern as its weekly write-ups — these are data-only edits
  to its own content); roster changes remain PR-gated.
- **E4 source:** aggregated from runs.json (last 7 days, per-bot counts of
  published outputs), not git log.

## Deep-Review Decisions (11-section review, all resolved)

- **D5/1A — loud trigger failure:** B4's radar parse validates entry shape; on
  malformed JSON or 0 readable files it logs status=error via log_run (visible
  on /pipeline) and exits non-zero. Silent no-proposals is impossible.
- **D6/2A + D12 — definitive-dead only, honest wording:** E3 auto-archives only
  on 404/410/DNS-fail/conn-refused on 3 CONSECUTIVE weekly checks. 403/429/
  timeout/5xx = "unverifiable": staged to ~/.softcat-bot-staging AND logged via
  log_run details (visible on /pipeline) — never auto-archived. Stamp wording is
  "link checked YYYY-MM-DD", never "verified" (a 200 doesn't prove alive).
  Checks are HTTP-status-only; fetched content never reaches the LLM.
- **D7/3A — tests:** `scripts/validate-tools-data.mjs` (models.json/manifest
  schema incl. trackedSince/radarRef + radarRef resolution; run MANUALLY before
  merging roster PRs, same workflow as validate-horizon-refs.mjs — deploy.yml
  untouched; a separate optional workflow file may come later) +
  pytest in bot/venv (`bot/tests/`) for the 4 pure functions: trigger gate,
  dead-link classifier, streak logic, rotation. No package.json changes.
- **D8/4A — pipeline identity:** both new Jobs log under existing ids
  (model_bot / tool_bot) with a `job` field ("roster" / "verify"). "Six bots"
  copy stays true (Hero.astro:23, pipeline.astro:73, horizon.astro:792).
- **D9/5A — single provenance surface:** B6 + E4 merge into ONE compact mono
  strip (per-source freshness + 7-day activity, absolute times and per-bot
  detail on hover/expand; expanded view links to the actual PRs/commits/pages,
  not bare counts — D15). Capability scores carry an "editorial scores" label
  in the UI tooltip (D15). E6 gets a zero-results state for filtered-to-empty
  (reuses the transmission-style empty state).
- Shadow-path guards (implementation requirements, not decisions): rotation
  skips spotlight when manifest is empty (mod-0 guard); corrupt
  tool_verify_history.json resets to empty with a logged warning; provenance
  strip renders "no data" for a bot missing from runs.json instead of failing
  the build; client age script falls back to absolute date on invalid
  timestamp; B5 flag removal + rotation land in the same PR.
